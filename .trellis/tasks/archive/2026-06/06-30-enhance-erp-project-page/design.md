# Enhance ERP Project Page Design

## Scope

This child task enhances the Ozon ERP page in BIAU Port as a visitor-readable technical case study. It does not edit the ERP source repository, deployed ERP runtime, browser extension package, database, Ozon configuration, blog posts, or assistant API contracts.

## Presentation Direction

Use a "small-team commerce operations system" structure:

1. Product story: Ozon store operations, product collection, shop authorization, product sync, pricing/profit tools, one-click listing, and operations traceability.
2. Admin workflow: Vue/Vite workspace with login/register, overview, shops, products, drafts, import history, tools, selection rules, product edit, and settings.
3. Backend architecture: Express API, Prisma/PostgreSQL schema, route areas, auth/extension-key boundary, uploads/static assets, API health, and startup recovery.
4. Plugin and Seller bridge: Chrome MV3/WXT extension, ERP API proxying, token sync, Seller tab checks, Seller context/cookie read, cross-tab Seller requests, and download helper.
5. Protected writes and jobs: Ozon credentials, `PendingAction`, `JobQueue`, audit logs, mock/real adapter boundary, queue/inline modes, Redis/BullMQ option, local background recovery, and official readback.
6. Verification and operations: API/shared tests, route tests, smoke script, handoff checklist, launch readiness report, commission coverage report, extension release checks, and dependency-audit rationale.
7. Current limitations and future optimization.

The page should feel practical and operational: first explain why the system exists, then show how the admin console, API, database, plugin, jobs, and safety boundaries work together.

## Public Safety

Do not put the following in public UI or generated assistant knowledge:

- IP addresses, deployment host/user details, SFTP details, panel details, private ports, database URLs, Redis URLs, JWT secrets, refresh secrets, ERP encryption keys, extension keys, Seller cookies, Ozon Client-Id/API-Key, real account names/passwords, backup hashes, or raw private deploy commands.
- Long raw report tables containing SKU/category examples from operational reports. Summarize the engineering conclusion instead.

It is safe to mention broad technologies and public product facts: Vue 3, Vite, Express, Prisma, PostgreSQL, Redis, BullMQ, WXT, Chrome MV3, Ozon Seller API, protected writes, queue/worker mode, API health, extension release checks, handoff checklist, launch readiness, and conservative commission display.

The existing public ERP entry URL from `src/data/siteLinks.ts` may remain as an external link because it is already part of the site.

## BIAU Port Code Boundaries

Expected touched files:

- `src/data/portfolio.ts`
- `src/pages/ProjectDetailPage.tsx`
- `server/data/public-knowledge.json` generated by `assistant:index`

Only touch `scripts/generate-assistant-knowledge.ts` or `src/data/assistant.ts` if the existing projection cannot represent ERP accurately. No assistant response shape change is expected.

Likely no CSS changes are needed because Legal RAG and Xunqiu already use reusable `.project-case-study` styling.

Avoid touching unrelated assistant admin/backend code, blog content, navigation, global styles, or source ERP files unless validation exposes a direct dependency.

## Data Model

Use the existing reusable model:

```ts
Project.detailContent?: ProjectDetailContent
Project.assistantContext?: string[]
```

Recommended ERP groups:

- `overview`: product story and why this is an operations system, not only a dashboard.
- `workflow`: admin routes, shop/product/draft/import/tool/selection workflows, plugin collection flow.
- `architecture`: web/API/shared/extension packages, route areas, Prisma schema, auth and extension-key boundaries, protected writes and queues.
- `quality`: tests, smoke, handoff checklist, launch readiness, commission coverage, dependency audit rationale.
- `limitations`: conservative commission coverage, real-write risk, extension key/config hygiene, Seller/Ozon dependency, queue/worker operation, dev-tool audit advisories.
- `roadmap`: production hardening, secret rotation/config cleanup, observability, richer retry dashboards, commission coverage, extension release automation, permission/audit improvements.

ERP currently has `OzonErpProjectArticle` hard-coded in `ProjectDetailPage.tsx`. Prefer moving its useful content into `portfolio.ts` and removing the conditional renderer for `project.id === 'ozon-erp'`, so ERP follows the shared detail renderer. Preserve the external ERP link and architecture article link through `project.links`.
