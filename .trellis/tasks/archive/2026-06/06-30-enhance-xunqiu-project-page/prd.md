## Requirements

- R1 Evidence-backed content: concrete claims about Xunqiu implementation, architecture, deployment, validation, or roadmap must be backed by inspected source files or framed as future direction.
- R2 Public safety: do not include old/new IPs, phone numbers, test passwords, password hashes, signing credentials, local SDK paths, database connection details, R2/Render secrets, private file paths, or raw operational commands in public UI copy.
- R3 Project page depth: the Xunqiu page should cover:
  - product problem and migration story
  - legacy app / backend constraints
  - Android 64-bit rebuild strategy
  - modern backend architecture
  - API compatibility and old envelope preservation
  - media upload and short-video pipeline
  - deployment shape: static showcase, Render backend, PostgreSQL, R2
  - verification: test matrix, backend tests, smoke script, APK/archive strategy
  - limitations and future optimization direction
- R4 Visitor-readable case study: content should lead with the product and migration value, then expand into architecture, validation, limitations, and roadmap.
- R5 Maintainable model: use the reusable `Project.detailContent` and `assistantContext` model from `src/data/portfolio.ts` instead of adding more Xunqiu-only hard-coded JSX.
- R6 Assistant compatibility: generated public knowledge must remain compatible with existing assistant citation shape.
- R7 Visual consistency: follow existing `ProjectDetailPage` and `flow-pages.css` patterns; avoid redesigning all project pages.
- R8 Validation: run at least `npm.cmd run assistant:index`, `npm.cmd run lint`, `npm.cmd run build`, and attempt `npm.cmd run verify` unless blocked with exact evidence.
