# Enhance Xunqiu Project Page Design

## Scope

This child task enhances the Xunqiu page in BIAU Port as a visitor-readable technical case study. It does not edit Xunqiu source repositories, the standalone Xunqiu showcase site, APK files, backend deployment, or blog posts.

## Presentation Direction

Use a "mobile system migration case study" structure:

1. Product story: football community, teams, matches, social feed, short videos, and mobile workflow.
2. Migration problem: old Android client, old Java WAR backend, native/video/third-party SDK risk, and legacy deployment complexity.
3. Rebuild strategy: new 64-bit Android client, old/new client split, configurable backend host, and old envelope compatibility.
4. Backend modernization: Spring Boot 3, Java 17, PostgreSQL/Flyway, Render Docker deployment, R2-compatible storage, and safe stubs.
5. Media and short-video pipeline: file checks, cover extraction, multipart upload, R2 object storage, playback ratio handling, comments/likes.
6. Verification and deployment: test matrix, legacy-entry matrix, backend MockMvc/Testcontainers tests, smoke script, APK archive, static showcase.
7. Current limitations and future optimization.

The page should feel like an engineering case study for visitors: practical product problem first, implementation and architecture below.

## Public Safety

Do not put the following in public UI or generated assistant knowledge:

- IP addresses or raw backend service host details beyond already-public product/backend links in BIAU Port.
- Phone numbers, test passwords, password hashes, login tokens, signing key aliases/passwords, keystore paths, SDK paths, local paths, old server usernames, database URLs, R2 keys, Render dashboard details, or cloud provider secrets.
- Raw deployment commands that include account/host details.

It is safe to mention broad technologies and architecture: Android 64-bit client, Spring Boot 3, Java 17, PostgreSQL, Flyway, Render, Cloudflare R2, Cloudflare Pages, Docker, smoke tests, and compatibility tests.

## BIAU Port Code Boundaries

Expected touched files:

- `src/data/portfolio.ts`
- `src/pages/ProjectDetailPage.tsx`
- `scripts/generate-assistant-knowledge.ts` only if projection behavior needs adjustment.
- `src/data/assistant.ts` only if projection behavior needs adjustment.

## Data Model

Use the existing reusable model:

```ts
Project.detailContent?: ProjectDetailContent
Project.assistantContext?: string[]
```

Recommended Xunqiu groups:

- `overview`: product and migration story.
- `workflow`: user-facing mobile workflows and module coverage.
- `architecture`: old/new split, Android 64-bit client, backend modernization, API compatibility.
- `quality`: test matrix, backend tests, smoke script, APK archive and validation chain.
- `limitations`: safety stubs, pending regressions, free-tier/demo boundaries, old system risks.
- `roadmap`: next iterations.

Xunqiu currently has `XunqiuProjectArticle` hard-coded in `ProjectDetailPage.tsx`. Prefer moving its useful content into `portfolio.ts` and removing the conditional renderer for `project.id === 'xunqiu'`, so Xunqiu follows the shared detail renderer. Keep Ozon's custom article unchanged in this child task unless a small cleanup is necessary.
