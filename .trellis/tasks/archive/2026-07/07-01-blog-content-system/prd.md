# Blog content system

## Goal

建立一个长期可维护的博客栏目体系与内容生成流水线，让博客不再依赖一次性批量生成，而是能按栏目、模板、取证、模型协作、审稿和发布检查来稳定生产精致内容。

本任务先进入规划：明确栏目、内容边界、模型使用策略和可复用 skill 的范围。后续实现时直接迁移博客数据模型、筛选 UI、公开策展和相关脚本到新栏目体系，并创建 `blog-content-pipeline` skill 与首批模板；不保留旧博客分类作为兼容层。

## Background

- 上一任务 `06-30-blog-content-cleanup` 已把公开博客收窄为 9 篇精选文章，其余 87 篇隐藏待重写。
- 现有博客基础分类来自 `src/data/blogShared.ts`：`tech | project | news | resource | daily`，显示标签为“技术 / 项目 / 新闻 / 资源 / 日常”。这些分类与新的站点栏目语义不匹配，后续实现应直接迁移，不做长期兼容。
- 新增公开策展契约记录在 `.trellis/spec/frontend/state-management.md`：默认隐藏未策展文章，公开列表、详情、助手、sitemap 和 loader 必须走 public selector。
- 用户不希望现在就执着确定具体文章内容，也不希望博客全是相似的项目文章。
- 用户倾向的栏目方向：`AI 日报`、`知识积累`、`资源分享`、`项目总结`，并可能再加入少量其他栏目。
- `资源分享` 的含义是用户自己分享有价值的资源、工具、文章、模板、仓库等，不是自动生成“资源清单模板”。
- `项目总结` 需要避免和项目详情页重复；项目页展示项目，博客讲阶段复盘、关键问题、取舍、踩坑、演进或反思。
- 模型渠道可用但不应过度编排：GLM-5.2、DeepSeek V4 Pro、Gemini 3.1 Pro、Gemini 3.5 Flash 可作为候选，不要求全部参与。
- 用户担心单个中转站多模型高并发，因此默认模型调用应串行；如必须并发，应明确哪些模型走不同中转站。
- 网站已有中英文切换。博客栏目不需要中英文两套含义，而应为同一栏目提供中文主标题与英文副标题/辅助标识。

## Proposed Column Model

一级栏目建议先规划为：

- `知识积累 / Knowledge Notes`：长期可复用的技术理解与方法沉淀，不一定绑定项目。例：RAG 引用溯源、Agent 工程边界、前端 AI 工作台、部署验证、内容治理。
- `项目总结 / Project Notes`：围绕项目阶段、问题、架构取舍、失败修复、迭代路线展开，不复制项目详情页。
- `资源分享 / Resource Picks`：用户主动分享的工具、文章、仓库、模型、模板、课程、素材等，重点是“我为什么推荐 / 我怎么用 / 适合谁”。
- `AI 日报 / AI Daily`：未来高频栏目，记录模型动态、工具更新、行业案例和可试能力；暂时只规划，不作为第一阶段正文重点。
- `构建手记 / Build Log`：记录站点、工作流、内容系统、AI 助手和 Trellis 使用过程，更偏实践过程和系统演进。

## Requirements

- R1: 定义博客一级栏目体系，区分“站点栏目”和“文章类型”，避免把所有内容都塞进项目相关分类。
- R2: 为每个栏目写清楚内容边界、适合题材、不适合题材，并给出从旧 `BlogCategory` 到新 `BlogColumn` 的一次性迁移规则；迁移后不保留旧分类兼容层。
- R3: 明确 `项目总结` 与项目详情页的分工，避免重复项目页已有的功能、技术栈、演示入口等稳定信息。
- R4: 明确 `资源分享` 是用户主观分享与使用笔记，不是自动生成的资源模板。
- R5: 设计默认模型协作策略：优先“Codex 取证 + 单个强内容模型生成/改写 + Codex 审稿入库”，而不是每篇都多模型流水线。
- R6: 说明何时需要多模型对照、何时禁止并发、何时可以并发但必须分中转站。
- R7: 规划 `blog-content-pipeline` skill：它应固化流程、模板、检查清单和模型选择规则，但不应把具体模型调用写死。
- R8: 第一阶段实现范围应小而稳：先改栏目/模板/流水线，不急于批量生成新文章。
- R9: `构建手记` 作为独立一级栏目保留，但首批内容不需要强行填充；它用于承载站点、内容系统、AI 助手和工作流演进。
- R10: 栏目应支持中文主标题与英文副标题/辅助标识；语言切换可以影响展示语言，但栏目语义保持同一套。
- R11: 博客代码层面应直接以 `BlogColumn` 表达栏目，公开筛选、文章详情、项目延展阅读、助手标签、sitemap 和审计脚本都应使用新栏目字段。

## Acceptance Criteria

- [ ] `prd.md` 记录栏目体系、内容边界、模型策略、skill 范围和待决问题。
- [ ] 若进入实现前仍属复杂任务，补齐 `design.md` 和 `implement.md`。
- [ ] 栏目规划能解释现有 `tech/project/news/resource/daily` 如何一次性迁移到新栏目，并明确不保留兼容分类。
- [ ] 模型策略明确默认串行，且说明高并发风险下的并发限制。
- [ ] `blog-content-pipeline` skill 的职责边界清晰：流程和质量门禁优先，具体模型可配置。
- [ ] 不在本阶段批量发布 AI 日报或大量新博客。
- [ ] 实现计划覆盖博客数据类型、公开策展、筛选 UI、详情页标签、项目延展阅读、助手索引、sitemap 和审计脚本。

## Out Of Scope

- 批量重写 87 篇隐藏文章。
- 建 CMS 或数据库内容管理。
- 自动高频发布 AI 日报。
- 写死某个模型渠道或依赖单个中转站。

## Decisions

- D1: `构建手记` 作为独立一级栏目保留。
- D2: 博客栏目采用同一套语义，提供中文主标题和英文副标题/辅助标识，不拆成中英文两套分类。
- D3: 不保留旧 `BlogCategory` 兼容层。后续实现直接迁移到 `BlogColumn`，旧 `category` 字段、旧 `categoryLabels` 和旧筛选语义应从博客域移除或重命名。
- D4: 第一阶段先完成栏目模型、公开筛选、现有 9 篇公开文章的栏目归属、skill 与模板；不批量重写 87 篇隐藏文章。
