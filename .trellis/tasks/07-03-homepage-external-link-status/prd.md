# 主页外链可用性检测与状态展示页

## Goal

把主页 IN PORT 项目卡片背后的公开入口整理成可检测、可展示的站点状态页，让访客能清楚看到主要外链入口当前是否可访问、响应速度大致如何、哪些入口只是演示或可能冷启动。

这个任务服务于长期目标里的“访客可见的产品体验与展示闭环”：主页卡片本身进入项目详情，卡片按钮打开项目网站 / demo / 工作台；状态页应解释这些外部入口的可用性，而不是只在代码或 CI 输出里隐藏失败信息。

## Requirements

- 基于仓库证据而不是手写猜测：
  - 主页 IN PORT 外部入口来自 `src/data/hero.ts`。
  - 项目详情更多外部链接来自 `src/data/portfolio.ts`。
  - 已有 `scripts/check-site-monitor.mjs` 可以抓取核心路由、抽取链接、检查 HTTP 状态和耗时。
- 新增一个访客可读页面，用于展示：
  - 最近检测时间、总体状态、检测基准站点。
  - 主页项目外链的状态列表，至少覆盖 Legal RAG、ERP、Playlab、寻球。
  - 状态含义：可用、异常、未检测 / 无数据、可能冷启动或受登录保护。
  - 每个入口保留跳转按钮，但不公开后台账号、密码、密钥、真实私有地址或部署后台信息。
- 检测数据不应依赖浏览器跨域探测；前端只读取公开 JSON / 静态数据。
- 检测脚本应减少误报：
  - 不把 `fonts.googleapis.com/`、`fonts.gstatic.com/` 这类资源域名根路径误认为访客项目入口失败。
  - 外部项目状态应来自显式定义的 showcase links，而不是页面内所有 `href/src` 混扫结果。
- 页面应符合现有 React / Vite / Semi Design / 全站视觉体系：
  - 路由应接入 `src/App.tsx`。
  - SEO 与 sitemap 应同步。
  - UI 适配移动端，不出现文本溢出或重叠。
- 不自动部署，不公开任何需要人审核的真实凭据或私有配置。

## Acceptance Criteria

- [x] 新增 `/status` 或等价公开路由，访客可看到主页项目外链状态。
- [x] 新增或扩展检测脚本，能生成前端可读取的公开状态数据，并可在本地命令行运行。
- [x] 状态页优先展示显式项目入口，不因字体资源根路径 404 把整体状态误判为项目不可用。
- [x] 页面提供外链跳转，但不泄露真实账号、密钥、生产后台地址、数据库连接串或私有运维信息。
- [x] sitemap / SEO 与新状态页一致。
- [x] 通过 `npm.cmd run lint`。
- [x] 通过 `npm.cmd run build`。
- [x] 通过 `git diff --check`。
- [x] 通过变更范围敏感信息扫描。

## Notes

- 线上巡检证据：`npm.cmd run site:monitor -- --check-external --max-links=40 --timeout=8000 --json` 显示核心路由均 200；失败项是字体域名根路径 404，实际字体 CSS 请求 200。这是脚本噪声，不应展示成项目入口失败。
- 当前用户提供的 GLM-5.2 限时 key 属于助手部署环境配置，不进入本任务文件、PRD、journal 或提交记录。
- 2026-07-03 澄清：用户真正想长期实现的是“每个项目的可控程度 / 功能健康”，例如 Legal RAG 的模型 API、RAG 问答和合同审查链路是否正常。当前任务收束为第一层“公开入口状态”；深度功能健康需要后续独立任务设计合成检查、项目侧安全 health/smoke 端点、Prometheus/Grafana 或等价观测栈。
