# Design

## Data Boundary

The implementation should stay in the portfolio data layer:

- `src/data/portfolio.ts` remains the source of truth for project cards, detail sections, and assistant summaries.
- `ProjectDetailContentSections` and the existing project detail page should render the new content without project-specific JSX.
- `server/data/public-knowledge.json` should be regenerated by the existing assistant indexing script.

This keeps the change consistent with the Legal RAG, Xunqiu, and ERP work that already moved rich case content into structured project data.

## Content Strategy

### Biau Playlab

Use `biau-playlab` as the primary game case study. It should explain the platform-level work:

- Astro 5 content site with content collections for games, devlogs, and articles.
- Six Godot projects exposed as detail pages and Web play links.
- Static assets, screenshots, playable embeds, milestones, next steps, and devlog relations captured in frontmatter.
- Build and deployment quality gates: content audit, Astro build, dist link audit, Cloudflare Pages deploy, playable export checks, and public endpoint checks.
- Current limitation: the game/project showcase is much stronger than the broader blog/article archive, which may be optimized later.

### Individual Games

Keep the six individual game entries concise. Add `assistantContext` to each entry so the public assistant can answer project questions accurately:

- Tetris: classic scoring, combo/B2B, rogue upgrade experiment, responsive UI, touch bridge, screenshot regression, mobile touch follow-up.
- Next Spacewar: three-wave short mission, drift/armored targets, obstacles, combo/reward loop, result review, compact showcase direction.
- intespace: vertical auto-shooter roguelite, chapter/survival/boss modes, weapon tree, meta progression, playtest polish and balance follow-up.
- Raiden: vertical shooter slice, two chapters, boss phases, chain rewards, public demo, tuning and content follow-up.
- space-war: retro side-scrolling shooter, five sectors, boss, high score/result loop, Web/Windows release posture, audio/telemetry/balance follow-up.
- Spacewar II: vertical mobile shooter sequel, enemy variety, boss phases, multidirectional bullet patterns, pickups, bombs, short combo windows, result loop, mobile UX follow-up.

Lightweight detail sections may be added if they improve the page, but avoid six long duplicate case pages. The platform case should carry the broad architecture narrative.

### Pet Workspace

Use `pet-workspace` as a work-in-progress technical case study. It should explain the current system without overclaiming:

- Android app shell, default pets, desktop overlay, generation UI models, community API clients, and package import flow.
- Community API as the only app-facing backend gateway, with feed, approved pets, packages, wallet, check-in, import drafts, submissions, admin review, SLA, metrics, health, and fantasy-pet proxy behavior.
- Admin Review UI and Pet Generator adapter as internal/supporting surfaces.
- `fantasy-pet-rule` generation service as the server-side generation and QA pipeline, not app-embedded logic.
- Human review remains the only acceptance gate; machine QA and GenericAgent/Codex evidence are repair/review pressure, not promotion authority.
- Public artifact safety uses opaque `downloadId` and excludes internal paths, worker commands, prompts, provider configuration, and secrets.
- Current gaps: Android emulator E2E and live private deployment have substitute evidence; observability, runbooks, auth/tenant isolation, worker daemon maturity, queue operations, and production SLA remain ongoing.

## Public Safety

The main site content may describe architecture, routes, quality gates, and high-level deployment posture. It must not include:

- Real IPs, accounts, passwords, tokens, API keys, connection strings, provider base URLs, or exact private host names.
- Internal env file paths or private adapter config locations.
- Commands that imply public users can launch workers, agents, image generation, private ops, or deployment scripts.
- Local source workspace paths in public-facing copy.

Task PRD and design notes may keep local evidence paths because they are internal planning artifacts.

## Compatibility

No schema migration is required. `Project.detailContent` already supports `overview`, `workflow`, `architecture`, `quality`, `limitations`, and `roadmap`. `assistantContext` is already included in `getProjectAssistantSummary`, and assistant indexing already reads the portfolio data.

## Rollback

Rollback is data-only:

- Revert the `src/data/portfolio.ts` changes for this task.
- Regenerate `server/data/public-knowledge.json`.
- Leave unrelated dirty files untouched.
