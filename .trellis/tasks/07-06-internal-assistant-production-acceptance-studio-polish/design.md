# Internal Assistant Production Acceptance and Studio Workflow Polish Design

## Boundaries

This task polishes the existing controlled `studio.draft` flow. It does not change the permission model:

- Internal `/assistant` can use read tools and `draft-write`.
- Studio review/export remains the only route to public content.
- Browser code never receives Studio admin credentials.
- No live model/provider diagnostics are added.

## Data Flow

```text
POST /chat/internal
  -> runInternalAgent()
  -> executeStudioDraft()
  -> ContentDraft created through getStudioPrisma()
  -> safe artifact returned in ChatResponse.meta.tools[]
  -> /assistant renders artifact link
  -> /studio?draft=<id> loads draft list and selects matching draft
```

## Artifact Contract

The safe Studio draft artifact remains intentionally narrow:

```ts
{
  kind: 'studio-draft'
  id: string
  slug: string
  title: string
  column: string
  status: 'review-needed'
  visibility: 'hidden'
  reviewRequired: true
  href: `/studio?draft=${string}`
}
```

`href` may also accept legacy `/studio` when normalizing older persisted messages. It must never include admin tokens, member tokens, API base URLs, body text, raw Prisma payloads, database identifiers, provider details, or private endpoints.

## Frontend Selection Behavior

`StudioPage` should read a safe query parameter such as `draft`. After drafts load:

- if a matching `id` exists, select it and show a short status message;
- optionally allow matching by `slug` for resilience if the link was generated by older tooling;
- if not found, keep the current selected/new draft state and show a non-blocking status;
- if token/API config is missing, keep existing token/config prompts.

The query parameter is navigation state only. It does not grant access and does not replace Studio token authentication.

## Diagnostics

Backend trace already has `status` and `errorClass`. Frontend should map those into member-readable hints:

- `not_configured`: Studio database is not configured for this service.
- `policy_blocked`: content looked sensitive and was not stored.
- `tool_error`: draft write failed and fell back to a plan.
- selected/completed with no artifact in plan-only mode: no database write happened.

Keep the detailed raw summary available only as safe text already produced by the server.

## Studio Review Cues

Drafts with `aiAssistance === "agentic-workspace"` should have a small visible marker in the draft list and/or editor header. This is an audit cue, not a trust claim. It must not imply the draft is ready to publish.

## Compatibility

- Older messages with `href: '/studio'` remain renderable.
- Existing `ContentDraft` schema is unchanged.
- Existing Studio token flow is unchanged.
- Existing smoke scripts remain no-live.

## Rollback

If deep-link behavior causes issues, revert the artifact `href` to `/studio` and remove query selection logic. Created drafts remain ordinary Studio drafts and can still be reviewed manually.
